  <name>Task 1: WordPress Health Check and Data Extraction</name>
  <files>scripts/wordpress-scraper.ts, data/raw-wordpress-export.json</files>
  <action>
    BEFORE scraping, perform WordPress health check:
    1. Verify WordPress site is accessible (HTTP 200)
    2. Verify WP REST API is enabled (/wp-json endpoint responds)
    3. Test authentication if required
    4. Log WordPress version and available endpoints

    If health check fails, STOP and report error - do not proceed with partial data.

    Create WordPress scraper script that:
    - Fetches all artists/team members from WordPress
    - Fetches all portfolio items with media URLs
    - Fetches all categories/tags
    - Handles pagination (get ALL records, not just first page)
    - Includes image URLs for later download
    - Logs progress and totals

    Save raw export with metadata:
    ```json
    {
      "exportDate": "2026-01-XX",
      "sourceUrl": "https://old-kinelab-site.com",
      "wpVersion": "X.X.X",
      "totals": {
        "artists": N,
        "portfolioItems": N,
        "categories": N,
        "media": N
      },
      "data": { ... }
    }
    ```
  </action>
  <verify>
    - `curl -s https://old-site/wp-json | jq .name` returns site name
    - `npx ts-node scripts/wordpress-scraper.ts` completes without errors
    - `data/raw-wordpress-export.json` exists and contains data
    - `jq '.totals' data/raw-wordpress-export.json` shows non-zero counts
  </verify>
  <done>WordPress content fully extracted with totals logged</done>
</task>

  <files>scripts/data-transformer.ts, data/transformed-data.json, data/expected-counts.json</files>
  <action>
    Create transformer script that:
    1. Maps WordPress fields to Payload CMS schema
    2. Converts image URLs to MinIO upload format
    3. Maps categories to Payload taxonomy
    4. Preserves relationships (artist -> portfolio items)
    5. Generates unique IDs for import

    CRITICAL: Generate expected-counts.json for verification:
    ```json

      "expectedCounts": {
        "artists": 5,
        "portfolioItems": 150,
        "categories": 12,
        "mediaFiles": 180
      },
      "checksums": {
        "artistNames": ["Name1", "Name2", ...],
        "portfolioTitles": ["Title1", "Title2", ...]
      }
    }
    ```

    This file is used in Plan 10-02 checkpoint to verify no data loss.
  </action>
  <verify>
    - `npx ts-node scripts/data-transformer.ts` completes without errors
    - `data/transformed-data.json` exists with Payload-compatible structure
    - `data/expected-counts.json` exists with counts matching raw export
    - `jq '.expectedCounts.artists' data/expected-counts.json` matches raw totals
  </verify>
  <done>Data transformed to Payload schema with expected counts documented for verification</done>
</task>

<task type="auto">
  <name>Task 3: Validate Transformation Integrity</name>
  <files>scripts/validate-transform.ts</files>
  <action>
    Create validation script that:
    1. Compares raw export counts to transformed counts
    2. Verifies no records were dropped during transformation
    3. Checks for required fields (title, slug, at least one image)
    4. Reports any data quality issues (missing fields, broken references)
    5. Generates validation report

    Validation must FAIL if:

    - Any artist has zero portfolio items (broken relationship)
    - Any portfolio item missing required title or slug

  </action>

    - Validation report shows 100% record retention
    - No data quality warnings in output
  </verify>
  <done>Transformation validated - zero data loss confirmed</done>
